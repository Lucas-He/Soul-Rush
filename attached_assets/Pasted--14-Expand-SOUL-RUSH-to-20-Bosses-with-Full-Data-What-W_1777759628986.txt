#14 - Expand SOUL RUSH to 20 Bosses with Full Data
What & Why
Expand the game from 10 bosses to 20 bosses. Introduce a rich, data-driven BOSSES array where each boss carries full metadata (name, title, difficulty, theme colors, personality, design description, arena modifier, reward item pool, and a waves array). Each wave object carries all fields needed to describe and execute it: id, name, description, duration, attackType, warningType, bulletSpeed, spawnRate, damage, arenaEffect, patternTags, and an execute string that maps to an attack function. Add a parallel ITEMS array of 20 collectible items, each with full metadata. The UI should derive wave counts from boss.waves.length dynamically.

Done looks like
The game has 20 distinct bosses, each with its own intro, theme colors, attack sequence, and post-boss screen.
Each boss correctly has the exact number of named waves specified (Boss 1 = 5, Boss 2 = 6, … Boss 20 = 25).
Every wave listed by name in the spec exists as a real wave object in the boss data AND has a working attack implementation that fires bullets/lasers/effects.
A BOSSES array and ITEMS array are declared at the top of SoulRush.tsx (TypeScript, not script.js) with all required fields per the spec.
The UI (boss intro screen, boss-win screen, title) reads boss.waves.length to display the wave total rather than hardcoding it.
All 20 items exist in the ITEMS array with every required field (id, name, type, description, maxStack, usable, allowedInLeaderboard, iconShape, themeColor, effect, bossSource, rarity, useFunction).
TypeScript has zero errors after the change.
All 10 existing bosses are preserved exactly, with their attack behaviors unchanged; new bosses 11–20 each have at least 3 handcrafted signature attack functions and the remainder filled with remixed variants.
Out of scope
Visual item inventory or item use UI (future task)
Leaderboard integration (Task #11)
Tutorial (Task #12)
Sound (Task #13)
Changing the canvas/render engine or splitting into multiple files
Steps
Extend the BossConf and Wave interfaces — Add title, difficultyLabel, themeColors, personality, design, arenaModifier, rewardItemPool, and waves: Wave[] to the existing BossConf TypeScript interface. Add a Wave interface with all required fields. Keep the existing attacks: string[] field — it is derived from waves.map(w => w.execute) so existing attack dispatch logic continues to work unchanged.

Expand BOSSES array entries 1–10 — Update each existing boss entry to include the new metadata fields and a properly structured waves array. Boss 1 gets 5 waves (Crystal Rain, Mirror Walls, Shatter Pulse, Prism Crossfire, Prism Prison); Bosses 2–10 get their specified wave names matching the existing attack functions already implemented. The attacks array should be derived automatically from waves.map(w => w.execute).

Add ITEMS array — Declare the ITEMS array of 20 items (Prism Apple, Glitch Cookie, Halo Thread, Clock Tea, Crown Shard, Deep Battery, Bone Charm, Star Jam, Engine Core, Rule Eraser, Velvet Antidote, Choir Bell, Gravity Pin, Ember Mint, Static Patch, Atlas Crumb, Ink Fold, Echo Pearl, Moon String, Mistake Heart) with all required fields per spec.

Add bosses 11–20 data entries — Write the full boss data objects for bosses 11–20 including all required metadata and wave objects. Boss 11 (15 waves), Boss 12 (16), Boss 13 (17), Boss 14 (18), Boss 15 (19), Boss 16 (20), Boss 17 (21), Boss 18 (22), Boss 19 (23), Boss 20 (25). Wave names per boss are specified in the plan. The execute string in each wave must map to one of the attack function names defined in step 5.

Implement attack functions for bosses 11–20 — Write at least 3 distinct handcrafted attack functions per new boss (signature waves), plus shared/remixed variants for the remaining waves. Each attack function must have: a warning phase (visual marker before bullets appear), an active phase with real bullet/laser/entity spawning, and a cooldown. Reuse existing attack primitives (rings, clockHands, pistonBlocks, diagLasers, etc.) and vary parameters to create distinct feel. Implement all 10 new boss attack dispatch blocks in updateAttack().

Wire wave count into UI — Update the boss intro screen, boss-win screen, and title screen to display boss.waves.length instead of any hardcoded wave count. Ensure the wave progress indicator (if any) uses the live count from the waves array.

Typecheck and verify — Run pnpm --filter @workspace/soul-rush run typecheck with zero errors. Use the admin panel (password: Orcas@0112, guide: solution0112) to jump to each new boss and confirm attacks fire without runtime errors.
